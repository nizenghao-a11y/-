# 练对页面雏形 V1

## 一、当前版本定位

当前已跑通的 UI 原型只代表 `练对` 业务版块的雏形，不代表完整小程序最终底部 Tab 架构。

当前原型的全局底部入口先只保留：

- `练对`
- `我的`

其中 `练对` 是当前重点打磨的业务版块，`我的` 是全局账户中心基础原型。

当前 `练对` 内部包含：

- `首页`
- `发单`
- `接单`
- `订单`
- `案例`

这些都是 `练对` 版块内部的业务页面或流程入口，不是最终全局底部导航。

后续完整小程序的底部 Tab 仍应回到：

- `练习`
- `练对`
- `论坛`
- `我的`

## 二、当前练对雏形包含的能力

### 1. 练对首页

用于承载动作点评业务的主入口。

当前包含：

- 动作点评主价值表达
- 上传动作 -> 选择套餐 -> 真人接单 -> 验收结算流程
- 适合先做的需求分类
- 我的练对订单状态入口
- 发单、接单、订单、案例四个内部入口
- 案例广场与广告位预留
- 当前卡点标注

### 2. 发单流程

用于模拟用户发布动作点评订单。

当前包含：

- 动作视频上传占位
- 问题类型选择
- 点评套餐选择
- 套餐交付标准
- 隐私订单 / 公开案例选择
- 托管金额并发布的交互占位

### 3. 接单流程

用于模拟点评人查看订单与接单。

当前包含：

- 可接订单列表
- 订单价格与类型展示
- 接单资格提示
- 点评提交模板
- 禁止私信、联系方式和导流的规则提示

### 4. 练对内的订单页

当前练对内的 `订单` 是练对业务内部的订单、退款、资质概览，不是最终全局 `我的` Tab。

当前包含：

- 订单状态线
- 部分退款比例规则
- 点评人资质审核提示

### 5. 全局我的页基础原型

当前 `我的` 是完整小程序的全局账户中心基础原型。

当前包含：

- 个人名片
- 账户与审核入口
- 钱包入口
- 客服设置入口
- 跨业务订单汇总
- 我的发布扩展占位

## 三、当前已确认的产品原则

### 1. 私信剔除

练对业务不做私信聊天，不做粉丝关系，不做陌生人社交。

允许保留：

- 订单内补充说明
- 平台客服
- 仲裁沟通

不允许保留：

- 自由私聊
- 联系方式交换
- 私域导流
- 粉丝关注体系

### 2. AI 不作为核心卖点

当前版本不做 AI 动作分析，不以 AI 纠错作为用户信任核心。

后续如引入，只能作为：

- 内部辅助标注
- 点评人效率工具
- 非决定性参考

### 3. 真人点评是核心

练对的核心价值是：

用户用较低成本，获得一次隐私、客观、可申诉的真人动作点评。

### 4. 套餐必须绑定交付标准

套餐不是单纯价格分层，而是交付标准分层。

示例：

- 基础点评：指出主要问题与简单建议
- 标准点评：提供多条观察、发力建议和辅助动作
- 人工精修：逐帧人工分析、风险提示和阶段建议

### 5. 退款支持部分退款

退款不只做全额退款，后续应支持按比例仲裁。

建议比例：

- `0%`：交付完整
- `30%`：内容偏浅
- `50%`：缺失关键交付项
- `100%`：敷衍、无关、违规

## 四、后续整合方向

后续完整小程序应这样拆：

### 1. 练习

独立承载动作库、器械使用教学、标准动作、基础知识。

### 2. 练对

承载当前这版雏形里的全部交易闭环：

- 发单
- 接单
- 点评交付
- 订单验收
- 退款仲裁
- 结算

### 3. 论坛

承载技术问答、案例广场、动作讨论和后期广告收入。

不做泛社交。

### 4. 我的

承载全局账户中心：

- 用户资料
- 资质审核
- 我的订单
- 钱包
- 客服
- 设置

## 五、不要跑偏的边界

后续继续设计时，不要把当前 UI 当成完整小程序结构。

当前 UI 只应被视为：

`练对版块 V1 业务雏形`

后续要做的是：

- 先保留这版练对业务链
- 再单独补齐练习、论坛、我的
- 最后把练对内部流程整合进正式底部 Tab 中的 `练对` 页面
